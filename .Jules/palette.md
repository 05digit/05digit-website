## 2024-03-24 - Semantic Audio Controls
**Learning:** Custom interactive components like a vertical volume slider and custom tracklist rows require careful handling to match native accessibility. While the volume bar needed `role="slider"`, `tabIndex={0}`, ARIA value attributes, and custom keyboard event handling (`onKeyDown` for arrow keys) because it acts like an `<input type="range">`, the tracklist rows could simply be switched from `<div>` to `<button>` to inherit native keyboard focus and interaction behavior for free.
**Action:** Default to using native `<button>` or `<input>` elements whenever possible to inherit keyboard semantics. When custom elements like `<div>` must be used for sliders, always implement `role="slider"`, full `aria-value*` bindings, and explicit arrow key handling to ensure keyboard usability matches pointer usability. Add `focus-visible` to all interactive components.
## 2024-05-18 - Missing Focus States on Custom Controls
**Learning:** Even when using native `<button>` elements, custom media controls heavily styled with Tailwind (e.g. `SkipBack`, `SkipForward`, `WATCH TRAILER`, and `MORE MUSIC` buttons) can lose explicit keyboard focus indicators if standard browser outlines are hidden or their contrast against dark backgrounds is poor. Keyboard users were unable to visually track their current focus when tabbing through the media player controls.
**Action:** Always verify that interactive elements not only have keyboard accessibility via semantics (`<button>`) but also visual indication when focused. Add `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-[#ff003c]` (or the brand accent color) to ensure clear, high-contrast focus rings for keyboard navigation.
